# 異常検知のメタ視点：分析前に立てるべき問い

---
## 1. 何の「形」の異常を捉えたいか

| 異常の形 | 内容 | 向いている手法 |
|---|---|---|
| スパイク | 1点だけ急に外れる | 3σ、IQR |
| レベルシフト | 水準がガクッと変わる | CUSUM、t検定 |
| ドリフト | じわじわ傾いていく | CUSUM、EWMA |
| パターン変化 | 値は正常だが周期性が乱れる | STL残差、スペクトル分析 |

---

## 2. いつの異常を捉えたいか

- **すでに起きている異常**を検知 → 閾値超えで即検知
- **進行中のトレンド**を早期検知 → 累積偏差を見るCUSUMが有効
- **予兆（まだ正常値の範囲内）**を検知 → 過去の故障データと照合が必要（教師あり寄りになる）

> 予兆検知は「異常からのズレ」ではなく「故障前特有のパターン」を捉えるため、
> ラベル付き故障履歴データがあるかどうかで実現可能性が変わる。

---

## 3. 検知後に何をするか（介入の設計）

- **自動停止** → 誤報が許されない → 閾値を緩めに設定
- **人間が確認して判断** → 多少の誤報はOK → 閾値を厳しく設定
- **ログに残すだけ** → 精度より網羅性を優先

---

## 4. どのくらい前に気づく必要があるか（リードタイム）

```
介入に2時間かかる → 2時間前に検知できなければ意味がない
```

- 必要なリードタイムを先に決める
- それを満たせる手法・パラメータを選ぶ
- CUSUMはじわじわした変化に早期反応できるため、リードタイムが必要な場面に向く

---

## 5. 見逃し vs 誤報のトレードオフ

```
閾値を厳しくする → 見逃しが減る、誤報が増える
閾値を緩くする   → 誤報が減る、見逃しが増える
```

どちらのコストが大きいかはビジネス文脈次第：
- リコール・重大事故リスク → 見逃しのコストが大きい → 厳しめに
- 誤報による生産停止コストが大きい → 緩めに

---

## 分析ノートの最初に書くこと（テンプレ）

```
- この分析で何を判断しようとしているか
- 想定している異常の形（スパイク／ドリフト／レベルシフト等）
- いつ検知したいか（リアルタイム／予兆／事後）
- 検知後の介入はどう行われるか
- 見逃しと誤報どちらを重視したか、その理由
```
